07 beat 2 · correct — the fall, then the method
the fall · “it looked perfect”

It compiled. The tests were green. It looked perfect. Then I checked against real data: a sort on a field most rows didn't even have had silently hidden 1,038,173 of 1,048,770 documents — 99% of a production queue. No error. No alert.

→ assume silent data loss until real row counts prove otherwise.

the catch · the system kept it honest

Then the fix nearly shipped — and it was worse: it would have taken the entire queue 100% offline and quietly stripped the repaired data back out. 17 independent review agents caught it against live production data, and found a cheaper fix touching 241 documents, not a million. Now a row-count check guards every query in CI.

→ the moat isn't the model. It's the harness that won't let a plausible fix ship unproven.

B4 backup · why this holds up in an enterprise

Not a chatbot config — a control plane

Governed the way enterprise code is governed: nothing ships on trust, everything ships on proof. (live: 15 skills · 5 reviewer agents · 9 hooks · 57 memory · 8 status · 619 subagents in 11 wks)

1 · safety scaled to blast radius

Prod data, secrets, and artifact security are gated by code that always runs — not the model's judgment. A deny-list blocks *prod* DELETE/DROP/UPDATE; secrets hit two independent guards (PreToolUse hook + git pre-commit).

2 · a transferable operating model

Mirrored 1:1 to Copilot, runs on any model, adopt piecemeal. The moat is the architecture, not my prompts — drop in the hooks day one, add a review gate week two.

3 · governance the org already speaks

Separation of duties (independent review agents) · least privilege (permission deny/ask lists) · audit trail (every change planned + proven on real data) · change control (no prod-flag flips, no force-push without a human).

why it worksIt maps onto the controls security & compliance already require — so it's defensible, not just fast.
